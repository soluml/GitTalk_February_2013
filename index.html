Why Responsive?

Why Responsive?

  • PC sales were expected to decline in 2012... for the first time since 2001.
  • Digitimes estimates that channel sales of tablets will grow 38.3% annually in 2013 to reach 210 million units, topping laptop PC shipments for the first time.
  • Whenever it happens... it's going to happen. Mobile traffic will overtake conventional PC traffic.

Arguments Against

Objective Arguments Against

  1. You hate change. #tablesrock
  2. You hate your clients. #moneysucks
  3. You hate your job. #I'mSorry...

Objective Arguments Against

It can confuse the user...

  • Responsive design sacrifices "desktop web" convention due to lacking screen real-estate. Side columns can shift or disappear. Navigations may reorganize. This can be confusing/frustrating.

*Solution*

  • Allow users to opt out and switch to the full site #opt-out. This is easier to implement if done from the get go! Also, don't sacrifice functionality if you don't have to... get creative!

Objective Arguments Against

It takes longer / costs more...

  • Generally speaking, responsive designs take longer to design and build than their non-responsive counterparts.
  • Non-responsive sites look fine on my phone. Why bother?

*Solution*

  • This can be true, but building a work of art often takes longer than building a piece of crap. You absolutely should convey the time/cost differences to your client, but, "it could take longer" isn't a good excuse to leave RWD out of your toolbelt. Give the solution most appropriate to the needs of the client.
  • Would it look better to your visitors if it was responsive? How does it work on a tablet? How about on a 72" LED TV? Responsive design scales both ways and accounts for any device regardless of screen size.

Objective Arguments Against

It's compromises design

  • In responsive design, the convention is to build "mobile first". This often leads to design compromises.

*Solution*

  • It doesn't have to. Your designer will have less screen real-estate to work with, but the power of media queries allows you to add design elements in as your screen gets bigger. Adaptive design also builds on this concept. Plus, we've given them the ability to opt out... right?

Should I use it?

Should I use it?

  • If your clients needs a web presence on multiple devices and can afford the extra cost of RWD, then choose a responsive design.
  • If not, sticking with a conventional design is probably the way to go.

How do we do it?

Media Queries!

Media Queries

  • In CSS, web developers can define a media type such as screen or print, and specify the look of the content by specifying conditions such as width, height, or orientation. A media query combines a media type and a condition to specify how web content will appear on a particular receiving device. #ThanksStanford
  • In layman's terms: Media Queries apply their CSS styles when their conditions are matched!
  • Support is fantastic for mobile browsers.
  • Examples | Detailed Information

Media Queries

Separate Sheet Style:

<link rel="stylesheet" media="screen and (min-device-width: 800px)" href="800.css" />

Inline Style:

@media screen and (min-device-width: 800px){
  //CSS goes here...
}
  • Media type definition comes first, in this case we're using "screen". Support of media types outside of Screen, Print, and All is lacking. #ListOfTypes
  • The above code will apply the media query styling to the document only if the device viewing it has a width of 800px or wider.

Media Query "Gotcha's"

Media Query "Gotcha's"

Media Queries aren't necessarily exclusive!

body {
    background: gray;
}
@media all and (min-device-width:500px) {
    body {
        background: blue;
    }
}
@media all and (min-device-width:700px) {
    body {
        background: red;
    }
}
  • The above code will produce a gray background by default. If the width of the screen is at LEAST 500px, the background will be blue... unless it's at least 700px... where it'll be red.
  • Old CSS rules still apply, beware of specificity and cascade order.

Media Query "Gotcha's"

There's a difference between width and device-width.

@media all and (min-width:700px){
  //CSS Here
}
@media all and (min-device-width:700px){
  //CSS Here
}
  • device-width: Describes the width of the output device (meaning the entire screen or page, rather than just the rendering area, such as the document window).
  • width: The width media feature describes the width of the rendering surface of the output device (such as the width of the document window, or the width of the page box on a printer).

Media Query "Gotcha's"

Max width or Min width?

  • Max means: "no more than". The window can't be any wider than "X" or these styles won't apply.
  • Min means: "no less than". The window can't be any thinner than "X" or these styles won't apply.
  • Min fits best with a mobile first approach. It lets you easily scale upward.

Media Query "Gotcha's"

PXs or %s or EMs?

  • Use EMs for anything text related: font-size, line-height, anything... #WhyEms?
  • Use percentages for structure like columns and grids.
  • Use PXs in your media queries, unless you want to adjust to the size of your content, then use EMs.

<meta name="viewport">

<meta name="viewport">

Why?

  • Mobile browsers assume that you're viewing a big desktop site so they set the viewport width (in the case of iOS Safari) to 980px.
  • This meta-tag rejects that assumption and adjusts the viewport to suit your needs.
  • Learn More

<meta name="viewport">

Typical Use Case for RWD

<meta name="viewport" content="width=device-width, initial-scale=1">
  • width=device-width: Instead of the 980px default, set the viewport width to the device width... let the media queries handle the rest.
  • initial-scale=1: Ensures that upon opening, your layout will be displayed properly at 1:1 scale. No zooming will be applied. This property can set a max/min zoom level or prevent zooming all together.

Adaptive Design

Adaptive Design

Adaptive vs Responsive

  • Responsive Design is Client Side. It's powered by media queries and is device agnostic.
  • Adaptive Design is Server Side. It detects the abilities of the device (primarily through its User Agent) and adjusts the content accordingly.
